
# INTERVIEW-QA

> 本文档整理了本项目可能涉及的面试问题及个人理解，主要帮助自己在介绍项目时能够更加清晰、有条理地表达自己的设计思路。

---

# 第一部分 项目整体

## Q1：请简单介绍一下你的项目。

### 回答

这是一个基于 C++ 实现的大语言模型统一接入 SDK。

项目参考课程内容完成，在学习 AI 大模型接口调用流程的基础上，对不同模型进行了统一封装，目前支持 DeepSeek、ChatGPT、Gemini 和 Ollama 等模型。

整个项目主要分为几个模块：

- ChatSDK
- LLMManager
- LLMProvider
- SessionManager
- DataManager
- ChatServer

其中 ChatSDK 对外提供统一接口，业务层无需关心底层调用哪个模型。

除了完成模型调用之外，我也尝试按照工程方式组织代码，例如模块划分、数据持久化以及会话管理等，希望能够让整个项目更加容易维护和扩展。

---

## Q2：为什么做这个项目？

### 回答

最开始只是希望学习如何通过 C++ 调用大语言模型接口。

后来发现不同模型虽然都提供聊天能力，但是调用方式存在很多差异。

如果业务代码直接调用不同模型接口，就会出现很多重复代码。

因此，我希望把这些公共逻辑抽象出来，实现一个统一的大模型接入 SDK。

通过这个项目，我不仅学习了 HTTP 和 JSON，也对 SDK 封装、模块划分以及工程化开发有了更加直观的认识。

---

## Q3：项目主要实现了哪些功能？

### 回答

主要包括：

- 多模型统一接入；
- ChatSDK 对外统一接口；
- 模型管理；
- 会话管理；
- SQLite 数据持久化；
- Web ChatServer 示例。

整个聊天流程都由 SDK 完成，业务层只需要调用 SDK 即可。

---

## Q4：整个项目有哪些模块？

### 回答

主要包括六个模块：

**ChatSDK**

统一入口，对外提供聊天接口。

**LLMManager**

负责管理所有模型 Provider。

**LLMProvider**

统一抽象接口，不同模型分别实现自己的 Provider。

**SessionManager**

维护聊天 Session。

**DataManager**

负责数据库相关操作。

**ChatServer**

SDK 的使用示例，负责接收 HTTP 请求。

---

## Q5：整个聊天流程是怎样的？

### 回答

整体流程如下：

用户发送消息

↓

ChatServer

↓

ChatSDK

↓

SessionManager 获取聊天上下文

↓

LLMManager 根据模型名称找到 Provider

↓

Provider 发送 HTTP 请求

↓

AI 模型返回回复

↓

更新 Session

↓

返回浏览器

整个流程中，每个模块都有独立职责。

---

# 第二部分 ChatSDK

## Q6：为什么需要 ChatSDK？

### 回答

如果没有 ChatSDK，那么业务代码需要：

- 创建 Provider；
- 管理 Session；
- 构造 HTTP 请求；
- 解析返回结果。

这些逻辑都会暴露给业务层。

增加 ChatSDK 后，业务层只需要调用统一接口即可，SDK 内部负责整个聊天流程。

这样业务代码更加简单，也方便后续维护。

---

## Q7：ChatSDK 的主要职责是什么？

### 回答

主要负责：

- 初始化整个 SDK；
- 管理聊天流程；
- 调用 SessionManager；
- 调用 LLMManager；
- 对外提供统一聊天接口。

它本身不直接发送 HTTP 请求，而是负责组织整个流程。

---

## Q8：为什么 ChatSDK 不直接调用 HTTP？

### 回答

因为 HTTP 请求属于具体模型实现。

不同模型：

- URL 不同；
- Header 不同；
- JSON 不同。

如果 ChatSDK 自己发送 HTTP 请求，就需要知道所有模型的实现细节。

因此把 HTTP 请求交给 Provider，实现职责分离。

---

## Q9：ChatSDK 最大的价值是什么？

### 回答

最大的价值就是：

**隐藏实现细节。**

业务层不知道：

- DeepSeek 怎么调用；
- Gemini 怎么调用；
- ChatGPT 怎么调用。

业务层只知道：

```cpp
chatSDK.sendMessage(...)
```

这也是 SDK 存在的意义。

---

# 第三部分 LLMManager

## Q10：为什么增加 LLMManager？

### 回答

如果没有 LLMManager，那么 ChatSDK 就需要维护所有 Provider。

例如：

```cpp
if(model=="DeepSeek")

else if(model=="Gemini")

else if(model=="ChatGPT")
```

随着模型越来越多，这种写法维护成本很高。

因此增加 LLMManager，由它统一管理所有 Provider。

---

## Q11：LLMManager 的职责是什么？

### 回答

主要包括：

- 注册 Provider；
- 初始化 Provider；
- 查询 Provider；
- 根据模型名称找到对应 Provider；
- 调用 Provider 完成聊天。

它相当于所有模型的统一管理者。

---

## Q12：新增一个模型需要修改哪些地方？

### 回答

整体步骤比较简单：

第一步：

新增一个 Provider。

例如：

```text
ClaudeProvider
```

第二步：

继承 LLMProvider。

第三步：

实现对应接口。

第四步：

注册到 LLMManager。

业务层基本不需要修改。

这也是采用统一接口最大的优势。

---

## Q13：为什么不用 switch 或 if-else？

### 回答

如果模型越来越多：

if-else 会越来越长。

不仅阅读困难，而且每增加一个模型，都需要修改已有代码。

采用 Provider + Manager 后，新模型只需要新增对应实现即可，扩展更加方便。


---

# 第四部分 LLMProvider

## Q14：为什么要设计 LLMProvider？

### 回答

这是整个项目最核心的设计。

虽然 DeepSeek、ChatGPT、Gemini、Ollama 来自不同厂商，但它们都提供了相同的能力——接收用户消息并返回模型回复。

因此，我将它们的共同能力抽象出来，定义了统一的 `LLMProvider` 接口。

业务层无需关心具体模型，只需要调用统一接口即可。

这样既降低了耦合，也方便后续继续扩展新的模型。

---

## Q15：LLMProvider 的职责是什么？

### 回答

LLMProvider 主要负责：

- 构造模型请求；
- 发送 HTTP 请求；
- 解析模型响应；
- 返回聊天结果。

它只关心"如何调用当前模型"，而不负责模型管理、Session 管理或数据库操作。

---

## Q16：为什么把不同模型拆成不同的 Provider？

### 回答

因为不同模型虽然功能类似，但接口实现差异较大，例如：

- 请求地址不同；
- Header 不同；
- API Key 配置方式不同；
- JSON 请求格式不同；
- JSON 响应格式不同。

如果全部写在一个类里，会出现大量 `if-else` 判断，代码可读性和可维护性都会下降。

拆分之后，每个 Provider 只负责自己的实现，更符合"一个类只负责一件事情"的原则。

---

## Q17：为什么使用抽象类，而不是普通工具函数？

### 回答

如果只是几个工具函数，那么业务层仍然需要自己决定调用哪个函数。

抽象类最大的优势在于：

所有 Provider 都遵循同一套接口。

这样 ChatSDK 和 LLMManager 都可以面向 `LLMProvider` 编程，而不用依赖具体实现。

后续增加新模型时，也不需要修改已有业务逻辑。

---

## Q18：LLMProvider 和 ChatSDK 有什么区别？

### 回答

两者职责不同。

ChatSDK 更偏向于业务流程组织，它负责：

- 管理聊天流程；
- 管理 Session；
- 调用模型；
- 对外提供统一接口。

LLMProvider 更偏向于底层实现，它只负责：

- 当前模型如何发送请求；
- 当前模型如何解析响应。

一个负责"组织流程"，一个负责"完成具体调用"。

---

## Q19：为什么 Provider 不负责管理 Session？

### 回答

因为 Session 属于聊天状态，而不是模型能力。

Provider 的职责应该尽量单一，只负责调用模型。

如果 Provider 同时保存聊天记录，那么它既负责模型调用，又负责状态管理，职责会变得混乱。

因此，我将聊天上下文交给 SessionManager 管理，Provider 只负责根据当前上下文生成回复。

---

## Q20：为什么 Provider 不直接操作数据库？

### 回答

数据库属于数据管理模块。

如果 Provider 直接操作数据库，就意味着模型调用和数据存储耦合在一起。

因此项目通过 DataManager 统一管理数据库。

Provider 只返回模型回复，是否保存数据由上层决定。

这样模块职责更加清晰。

---

## Q21：为什么每个模型都有独立的 cpp 文件？

### 回答

虽然不同模型流程相似，但实现细节不同。

如果所有模型都写在一个 cpp 文件中：

- 文件会越来越大；
- 阅读困难；
- 修改一个模型容易影响其他模型。

拆分之后，每个 Provider 都相互独立，也方便后续维护。

---

## Q22：如果新增 Claude 模型，需要修改哪些地方？

### 回答

整个流程比较简单：

第一步，新建 ClaudeProvider。

第二步，继承 LLMProvider。

第三步，实现统一接口。

第四步，在 LLMManager 中完成注册。

业务层基本无需修改。

这也是整个 Provider 架构最大的优势。

---

## Q23：为什么业务层不知道具体 Provider？

### 回答

业务层真正关心的是：

"我要和哪个模型聊天。"

而不是：

"这个模型 HTTP 怎么写。"

因此，Provider 的存在就是为了隐藏这些实现细节。

业务层只需要告诉 SDK：

```cpp
model = "DeepSeek"
```

SDK 会自动找到对应 Provider 完成后续工作。

---

## Q24：如果以后支持十几个模型，这种设计还能继续使用吗？

### 回答

可以。

因为新增模型不会影响已有 Provider。

只需要：

新增一个 Provider；

实现统一接口；

完成注册。

整个架构可以继续保持不变。

当然，当模型数量进一步增加时，也可以考虑增加配置管理或自动注册机制，但目前项目规模下，这种方式已经能够满足需求。

---

## Q25：你认为 Provider 设计最大的优点是什么？

### 回答

我认为主要有三个优点：

第一，降低耦合。

业务层不依赖具体模型实现。

第二，方便扩展。

新增模型不需要修改已有业务代码。

第三，职责清晰。

每个 Provider 只负责一个模型，维护起来更加简单。

完成这个模块之后，我对"抽象接口"和"面向接口编程"有了更加直观的理解。
