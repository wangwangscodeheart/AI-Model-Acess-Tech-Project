# AI Model Access Tech
## 项目概览
本项目是我在学习 **C++ 工程开发** 和 **AI 大模型接口调用** 过程中完成的一个实践项目。

项目通过**统一抽象接口**，对不同的大语言模型进行了封装，并实现了模型管理、会话管理、数据持久化以及 Web ChatServer 示例程序。

整个项目更加关注 **代码组织方式** 和 **模块设计**，而不仅仅是完成一次 HTTP 请求。

目前支持：
- DeepSeek
- ChatGPT
- Gemini
---
## 项目特色
- 使用统一 Provider 接口封装不同模型
- 使用 ChatSDK 对外提供统一调用接口
- 支持多模型切换
- 支持多轮聊天
- 使用 SQLite 保存聊天数据
- 提供 Web ChatServer 示例
- 使用 CMake 管理整个工程

## 项目简介

AI Model Access Tech 是一个基于 C++ 实现的大语言模型统一接入 SDK。

在学习 AI 大模型接口调用流程的基础上，对不同模型进行了统一封装，并实现了模型管理、会话管理、数据持久化以及 Web ChatServer 示例程序。

目前支持接入多个主流大语言模型，包括：

- DeepSeek
- ChatGPT
- Gemini

整个项目采用**模块化设计**，将模型调用、会话管理、数据管理等功能进行了拆分，使各模块职责更加清晰，也方便后续继续扩展新的模型。

---

## 项目背景

随着大语言模型的发展，不同模型厂商都提供了各自的 API 接口，但它们在请求地址、认证方式以及请求参数等方面存在一定差异。

如果业务代码直接调用不同模型接口，就需要分别处理不同的 HTTP 请求和 JSON 数据，代码容易产生重复，也不利于后续维护。

因此，本项目参考 SDK 的设计思想，对不同模型进行了统一封装，对外提供一致的调用接口，使业务层只关注"发送消息"这一行为，而无需关心底层调用的是哪个模型。

通过这个项目，我主要希望学习以下内容：

- C++ 工程组织方式
- SDK 封装思想
- HTTP 网络通信
- JSON 数据解析
- 面向对象设计
- CMake 项目管理
- SQLite 数据持久化

## 项目功能

目前项目主要实现了以下功能：

### 多模型统一接入

项目通过统一的 Provider 接口封装不同模型，目前支持：

- DeepSeek
- ChatGPT
- Gemini

业务层只需要指定模型名称即可完成调用，无需修改其他代码。

---

### SDK 封装

项目将模型调用过程统一封装到 ChatSDK 中。

ChatSDK 对外提供统一接口，包括：

- 初始化模型
- 创建聊天会话
- 获取聊天会话
- 查询模型列表
- 发送聊天请求
- 流式聊天接口

业务层无需直接操作 HTTP 请求。

---

### 会话管理

项目实现了 SessionManager，用于管理聊天会话。

每个 Session 保存独立聊天历史，使不同聊天之间互不影响，也为后续支持多轮对话提供基础。

---

### 数据持久化

项目使用 SQLite 保存聊天相关数据。

数据访问统一交由 DataManager 管理，业务模块无需直接操作数据库，提高了代码的可维护性。

---

### Web ChatServer

项目提供了一个简单的 ChatServer 示例。

浏览器发送请求后，由 ChatServer 调用 ChatSDK，再由 SDK 调用对应的大模型接口，最终返回聊天结果。

整个过程展示了 SDK 在实际业务中的使用方式。

---

## 项目整体架构

整个项目主要分为三层：

```text
                ChatServer
                     │
                     ▼
                 ChatSDK
        ┌────────────┴────────────┐
        ▼                         ▼
 SessionManager                 LLMManager
                                      │
                                      ▼
                               LLMProvider
                                      │
					     ┌────────────┬────────────┬
				         ▼            ▼            ▼            
					   DeepSeek     ChatGPT       Gemini     
```

其中：

- ChatServer 负责接收客户端请求；
- ChatSDK 负责组织整个聊天流程；
- LLMManager 负责管理所有模型；
- LLMProvider 定义统一模型接口；
- SessionManager 管理聊天上下文；
- DataManager 负责数据持久化。

各模块职责相对独立，便于维护与扩展。

# 核心模块设计

## ChatSDK

ChatSDK 是整个项目对外提供的统一入口，也是业务层唯一需要直接使用的模块。

如果业务代码直接调用各个 Provider，不仅需要了解不同模型的接口格式，还需要负责模型选择、会话管理等工作，随着支持模型数量增加，业务代码会越来越复杂。

因此，项目将这些逻辑统一封装到 ChatSDK 中，对外提供一致的调用接口。

ChatSDK 主要负责：

- 初始化整个 SDK；
- 调用 LLMManager 完成模型管理；
- 调用 SessionManager 管理聊天会话；
- 对外提供统一聊天接口；
- 屏蔽不同模型之间的实现差异。

业务层只需要创建 ChatSDK 对象，即可完成整个聊天流程，而无需关心底层具体调用的是哪个模型。

这种设计也符合"高层模块不依赖底层实现"的思想。

---

## LLMManager

随着项目支持的大模型越来越多，如果每次聊天都直接创建对应 Provider，不仅代码重复，而且后续增加模型时需要修改大量业务代码。

因此项目增加了 LLMManager，专门负责所有模型 Provider 的统一管理。

主要职责包括：

- 注册所有 Provider；
- 初始化模型；
- 查询当前支持的模型；
- 根据模型名称找到对应 Provider；
- 调用指定模型完成聊天。

业务层无需关心具体模型类型，只需要传入模型名称即可。

例如：

```text
ChatSDK
      │
      ▼
LLMManager
      │
      ▼
根据模型名称找到对应 Provider
      │
      ▼
发送请求
```

这种设计降低了 ChatSDK 与具体模型之间的耦合，也方便后续继续扩展新的 Provider。

---

## LLMProvider

不同 AI 平台虽然都提供聊天能力，但它们的 API 格式并不完全相同。

例如：

- 请求地址不同；
- Header 不同；
- API Key 配置方式不同；
- JSON 请求格式不同；
- 返回数据格式不同。

如果直接把这些逻辑写到 ChatSDK 中，会导致大量 `if...else` 判断。

因此项目抽象出统一的 `LLMProvider` 接口。

每一个模型只需要实现自己的 Provider 即可。

例如：

```text
      LLMProvider
            ▲
            │
 ┌──────────┼───────┐
 │          │       │
 ▼          ▼       ▼
DeepSeek  ChatGPT   Gemini

```

这样做带来的好处包括：

- 不同模型之间相互独立；
- 统一调用接口；
- 新增模型无需修改业务代码；
- 后续维护更加方便。

这种设计体现了面向接口编程的思想，也提高了项目的扩展能力。

---

## SessionManager

聊天机器人与普通 HTTP 请求最大的区别在于需要保存上下文。

如果每次请求都重新发送，就无法实现连续对话。

因此项目增加了 SessionManager，用于维护聊天会话。

每个 Session 保存当前聊天历史，SDK 在调用模型之前，可以根据 Session 获取历史消息，再一起发送给模型，实现多轮对话。

SessionManager 主要负责：

- 创建 Session；
- 查询 Session；
- 删除 Session；
- 保存聊天历史；
- 获取聊天历史。

模型调用与 Session 管理分别由不同模块负责，使各模块职责更加清晰。

---

## DataManager

聊天过程中会产生大量聊天记录。

如果所有模块都直接操作数据库，不仅容易造成代码重复，也不利于后续维护。

因此项目将数据库相关操作统一封装到 DataManager 中。

主要负责：

- 数据库初始化；
- Session 数据管理；
- 聊天记录保存；
- 聊天记录查询。

SDK 只调用 DataManager 提供的接口，而无需关心底层 SQL 实现。

这种方式降低了业务逻辑与数据库之间的耦合。

---

## ChatServer

为了验证 SDK 是否能够正常工作，项目实现了一个简单的 ChatServer。

ChatServer 并没有实现模型调用逻辑，而是作为 SDK 的使用示例。

整个处理流程如下：

```text
浏览器

    │

HTTP 请求

    │

ChatServer

    │

ChatSDK

    │

LLMManager

    │

LLMProvider

    │

AI Model

    │

返回结果

    │

浏览器
```

可以看到，ChatServer 只负责接收请求和返回结果，真正的聊天逻辑全部由 SDK 完成。

这样的设计使 SDK 可以方便地集成到其他项目中，而不仅限于当前的 Web 示例。

# 整体调用流程

下面以一次普通聊天请求为例，说明整个项目的调用流程。

### 第一步：创建聊天请求

客户端输入聊天内容，并选择需要使用的大模型。

例如：

```text
模型：DeepSeek

内容：你好，请介绍一下自己。
```

随后 ChatServer 接收到该请求。

---

### 第二步：调用 ChatSDK

ChatServer 将用户输入交给 ChatSDK。

此时业务层不需要关心：

- HTTP 请求如何发送；
- JSON 如何组织；
- API Key 如何配置；
- 请求地址是什么。

这些细节都由 SDK 内部负责。

---

### 第三步：获取当前会话

ChatSDK 根据 Session ID 获取对应聊天会话。

如果当前 Session 不存在，则创建新的 Session。

如果存在，则读取历史聊天记录。

这样便能够实现连续对话。

---

### 第四步：选择对应 Provider

随后 ChatSDK 调用 LLMManager。

LLMManager 根据模型名称找到对应 Provider。

例如：

```text
DeepSeek

↓

DeepSeekProvider
```

如果选择 Gemini，则会自动调用 GeminiProvider，而业务代码无需修改。

---

### 第五步：发送 HTTP 请求

Provider 根据对应平台接口要求：

- 构造 JSON 请求；
- 设置 Header；
- 添加 API Key；
- 发送 HTTP POST 请求。

随后等待模型返回结果。

---

### 第六步：解析响应

收到响应后，Provider 对 JSON 数据进行解析。

提取模型生成的回复内容，并返回给 ChatSDK。

---

### 第七步：更新聊天记录

ChatSDK 将本次聊天内容更新到 Session 中。

如果启用了数据持久化，还会调用 DataManager 保存聊天记录。

这样下一次聊天时即可继续使用之前的上下文。

---

### 第八步：返回结果

最终，ChatSDK 将模型回复返回给 ChatServer。

ChatServer 再通过 HTTP 响应返回浏览器，完成一次完整聊天流程。

# 项目亮点

本项目不仅实现了多个大模型的统一接入，也根据软件工程的思想对整个项目进行了模块划分，相比于直接调用 HTTP 接口，更加注重代码的组织方式和可扩展性。

### 统一模型接口

不同厂商提供的大模型接口存在一定差异，例如请求地址、认证方式、请求参数以及返回格式等。

项目通过 `LLMProvider` 抽象接口对这些差异进行了统一封装，业务层始终通过相同的接口完成模型调用，避免了大量重复代码。

这种设计也使得后续增加新的模型时，只需要新增对应 Provider，而无需修改已有业务逻辑。

---

### 模块职责划分清晰

整个项目按照功能划分为多个模块：

- ChatSDK
- LLMManager
- SessionManager
- DataManager
- ChatServer

各模块之间通过接口协作，每个模块只负责自己的职责。

这种划分方式也让我对"高内聚、低耦合"有了更加直观的理解。

### SDK 与业务解耦

ChatServer 并不直接调用模型接口，而是通过 ChatSDK 完成整个聊天流程。

因此，如果后续需要开发桌面程序、命令行程序或其他应用，只需要调用 ChatSDK 即可，无需重复实现模型调用逻辑。

### 支持多轮对话

项目通过 SessionManager 维护聊天历史。

每个 Session 保存独立的上下文信息，使聊天能够保持连续性，而不是每次请求都重新开始。

### 数据持久化

聊天数据统一交由 DataManager 管理。

数据库相关操作不会出现在业务代码中，使代码结构更加清晰，也方便后续扩展更多数据管理功能。

# 设计思路

在完成项目的过程中，我不仅关注功能是否能够实现，也尝试思考如何让代码更容易维护和扩展。

因此，在项目开发过程中，我主要遵循了以下几个设计思路。

---

## 为什么要封装 ChatSDK？

如果业务代码直接调用不同模型接口，那么每增加一个模型，都需要重新编写一套 HTTP 请求逻辑。

例如：

```text
业务代码

↓

DeepSeek HTTP

↓

Gemini HTTP

↓

ChatGPT HTTP

```

这样会导致：

- 业务层直接依赖模型接口；
- 不同模型存在大量重复代码；
- 新增模型需要修改业务代码。

因此，项目增加了 ChatSDK，对外提供统一接口。

业务层只需要调用：

```cpp
chatSDK.sendMessage(...)
```

至于底层调用哪个模型、如何发送 HTTP 请求、如何解析 JSON，都交由 SDK 内部完成。

这样能够降低业务代码与模型实现之间的耦合。

---

## 为什么设计 LLMProvider？

不同模型虽然接口不同，但都具有相同的功能：

- 发送消息；
- 获取回复。

因此，项目定义了统一的 `LLMProvider` 抽象接口。

每个模型只需要实现自己的 Provider。

例如：

```text
LLMProvider

├── DeepSeekProvider

├── ChatGPTProvider

└── GeminiProvider
```

这样做最大的好处是：

业务层始终面对统一接口，而不是具体模型。

以后如果需要增加新的模型，例如：

- Qwen
- Claude
- Kimi

只需要新增对应 Provider 即可，其他模块基本无需修改。

---

## 为什么增加 LLMManager？

项目支持多个模型后，如果每次聊天都自己创建 Provider，会出现大量重复判断。

例如：

```cpp
if(model=="DeepSeek")
{

}

else if(model=="Gemini")
{

}

else if(model=="ChatGPT")
{

}
```

随着模型数量增加，这种写法会越来越难维护。

因此，项目增加了 LLMManager。

LLMManager 统一维护所有 Provider。

聊天时：

```text
模型名称

↓

LLMManager

↓

找到对应 Provider

↓

发送请求
```

这样 ChatSDK 无需关心具体 Provider 类型。

---

## 为什么 SessionManager 单独管理？

聊天机器人最大的特点就是需要保存上下文。

如果每次聊天都重新发送消息，就无法实现连续对话。

因此项目将 Session 管理独立出来。

SessionManager 专门负责：

- 创建 Session；
- 保存聊天记录；
- 查询聊天历史；
- 删除 Session。

模型调用只负责生成回复。

Session 管理只负责维护上下文。

两者职责互不影响。

---

## 为什么 DataManager 独立出来？

聊天过程中需要保存大量数据。

如果数据库代码直接写在业务模块中，会导致：

- SQL 代码到处都是；
- 模块之间耦合较高；
- 后续修改数据库比较麻烦。

因此项目统一由 DataManager 管理数据。

业务模块无需直接操作 SQLite。

这种方式也方便以后替换其他数据库。

---

## 为什么 ChatServer 不直接调用模型？

ChatServer 的职责只是：

- 接收 HTTP 请求；
- 调用 SDK；
- 返回结果。

真正的聊天逻辑全部放在 SDK 中。

这样以后如果开发：

- Qt 客户端；
- 命令行程序；
- Linux 服务；

都可以直接复用 ChatSDK，而无需重新实现模型调用逻辑。

完成这个项目之后，我最大的收获并不是学会了调用 AI 接口，而是开始理解：

> **功能实现只是第一步，如何组织代码同样重要。**

相比于将所有代码写在一个类中，通过合理划分模块、抽象公共接口，可以让整个项目更加清晰，也更容易维护。

虽然目前这个项目规模还不大，但我希望在今后的学习过程中，能够逐步培养自己的工程化开发思维，而不仅仅停留在实现功能本身。
