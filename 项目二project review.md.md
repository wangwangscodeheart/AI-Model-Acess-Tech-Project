# PROJECT_REVIEW

# AI Model Access Tech 项目复盘

## 项目背景

这个项目完成于我学习 C++ 工程开发和 AI 大模型接口调用的过程中。

最初接触大语言模型时，我能够通过 HTTP 请求调用模型接口，实现最基本的聊天功能。但随着支持的模型越来越多，我发现不同模型之间虽然都提供聊天能力，却存在很多实现上的差异，例如：

- 请求地址不同；
- API Key 配置方式不同；
- 请求参数格式不同；
- 返回 JSON 格式不同。

如果每接入一个模型都重新编写一套调用逻辑，不仅代码重复，而且业务代码会越来越复杂。

因此，我希望能够将这些共同的功能抽象出来，实现一套统一的大模型接入 SDK，让业务层只关注"发送消息"这一件事情，而不需要关心底层具体调用的是哪个模型。

这也是完成本项目的初衷。

---

# 项目目标

完成项目之前，我给自己设定了几个比较明确的目标。

## 目标一：完成多个模型的统一接入

希望能够使用相同的调用方式完成：

- DeepSeek
- ChatGPT
- Gemini
- Ollama

等不同模型的调用，而不是为每一个模型分别编写业务代码。

---

## 目标二：学习 SDK 的基本封装方式

相比于直接编写一个聊天程序，我更希望尝试将模型调用封装成一个可以复用的 SDK。

这样以后无论开发：

- Web 服务；
- 命令行程序；
- 桌面程序；

都能够直接使用这一套接口。

---

## 目标三：尝试按照工程方式组织代码

项目不仅关注功能是否实现，也希望练习：

- 模块划分；
- 头文件与源文件分离；
- CMake 工程组织；
- 第三方库管理。

让整个项目更接近实际开发中的代码组织方式。

---

# 开发过程

整个项目的开发过程可以大致分为几个阶段。

## 第一阶段：完成模型接口调用

最开始，我首先完成了单个模型的 HTTP 调用。

这一阶段主要学习了：

- HTTP 请求发送；
- Header 设置；
- API Key 配置；
- JSON 请求构造；
- JSON 响应解析。

虽然已经能够完成聊天，但代码几乎全部集中在一起，可维护性较差。

这也让我开始思考，是否可以进一步抽象公共逻辑。

---

## 第二阶段：抽象统一 Provider

随着支持的模型越来越多，我发现每个模型虽然接口不同，但整体流程基本一致：

1. 构造请求；
2. 发送 HTTP 请求；
3. 解析返回结果；
4. 返回最终回复。

于是，我将这些共同能力抽象成 `LLMProvider` 接口，不同模型分别实现自己的 Provider。

这样既减少了重复代码，也使新增模型变得更加简单。

这是整个项目中我认为最重要的一次重构。

---

## 第三阶段：增加模型管理

有了多个 Provider 后，我发现业务层仍然需要判断当前应该调用哪个模型。

如果直接使用大量 `if-else` 判断，代码会随着模型数量增加越来越复杂。

因此，我增加了 `LLMManager`，负责统一管理所有 Provider。

业务层只需要指定模型名称，LLMManager 即可找到对应 Provider 并完成调用。

这样进一步降低了业务层与模型实现之间的耦合。

---

## 第四阶段：增加 Session 管理

完成模型调用之后，我开始尝试支持连续聊天。

为了保存聊天上下文，我将 Session 管理独立出来，由 SessionManager 负责维护每一个聊天会话。

这样模型调用只负责生成回复，而聊天记录则由 SessionManager 统一维护，模块之间职责更加明确。

---

## 第五阶段：增加数据持久化

为了避免聊天记录仅保存在内存中，我增加了 DataManager，并使用 SQLite 保存聊天相关数据。

数据库相关操作全部集中在 DataManager 中，其他模块无需直接操作数据库。

这一阶段让我对"业务逻辑"和"数据访问"分离有了更深的理解。

---

## 第六阶段：完成 ChatServer 示例

最后，我实现了一个简单的 ChatServer。

它主要用于演示 SDK 的实际使用方式。

ChatServer 本身并不负责模型调用，而是作为业务层，通过调用 ChatSDK 完成整个聊天流程。

通过这个示例，我也验证了 SDK 能够独立于具体业务进行复用。
---

# 开发过程中的思考

整个项目完成后，我发现真正花费时间的并不是实现 HTTP 请求，而是如何组织这些代码。

相比于"把功能做出来"，我更多地开始思考：

- 哪些代码应该放在一起？
- 哪些功能应该独立出来？
- 模块之间应该如何协作？

下面记录几个让我印象比较深的思考过程。

---

## 从"能运行"到"方便维护"

项目刚开始时，我更关注的是如何成功调用大模型接口。

只要能够发送 HTTP 请求，并正确解析返回结果，我认为项目就完成了。

但随着支持的模型越来越多，我发现很多代码开始不断重复。

例如：

- 每个模型都需要发送 HTTP 请求；
- 每个模型都需要构造 JSON；
- 每个模型都需要解析返回结果；
- 每个模型都需要处理认证信息。

虽然这些实现细节不同，但整体调用流程非常相似。

这让我意识到，如果继续复制代码，项目会越来越难维护。

因此，我开始尝试将共同逻辑抽象出来，而不是简单复制已有代码。

---

## 为什么没有把所有逻辑写在 ChatSDK 中？

刚开始设计时，我也考虑过直接让 ChatSDK 完成所有事情。

例如：

- 管理模型；
- 保存 Session；
- 发送 HTTP 请求；
- 解析 JSON；
- 操作数据库。

这种方式实现起来比较直接，但当功能逐渐增加时，一个类会承担越来越多职责。

后来我尝试把不同功能拆分出来：

- 模型管理交给 LLMManager；
- 会话管理交给 SessionManager；
- 数据管理交给 DataManager；
- 模型调用交给各个 Provider。

拆分之后，每个模块只负责自己的功能，代码结构也更加清晰。

---

## Provider 的抽象让我理解了"统一接口"

项目最大的变化，就是增加了 LLMProvider。

刚开始我只是觉得：

"不同模型写不同类就可以了。"

后来真正完成之后，我才意识到：

真正重要的不是"写几个类"，而是让这些类遵循同一套接口。

这样 ChatSDK 并不需要知道当前到底是 DeepSeek、Gemini 还是 ChatGPT。

它只需要调用统一接口即可。

这也是整个项目让我收获最大的地方之一。

---

## Session 为什么独立管理？

最开始，我准备把聊天记录直接保存在 Provider 中。

后来发现这种设计存在两个问题：

第一，Provider 的职责应该是负责模型调用，而不是保存聊天状态。

第二，如果以后更换模型，希望聊天记录仍然能够继续使用，那么 Session 不应该依赖具体 Provider。

因此，我最终选择将 SessionManager 单独拆分出来。

这样聊天状态和模型实现彼此独立，职责更加明确。

---

## 数据访问为什么单独封装？

项目中使用 SQLite 保存聊天数据。

如果所有模块都直接编写 SQL，不仅代码容易重复，也会导致数据库操作分散在整个项目中。

因此，我增加了 DataManager。

所有数据库相关操作都统一由 DataManager 管理。

这样业务层只需要调用对应接口即可，不需要关心底层数据库实现。

虽然目前项目规模还不大，但这种组织方式让我对数据访问层有了初步认识。

---

# 开发过程中遇到的问题

在完成项目的过程中，也遇到了一些值得记录的问题。

---

## 不同模型接口存在差异

虽然不同厂商都提供聊天能力，但接口细节并不完全一致。

例如：

- 请求地址不同；
- Header 配置不同；
- 请求 JSON 格式不同；
- 返回 JSON 结构不同。

因此，我没有尝试写一个"万能请求函数"，而是将这些差异放到各自 Provider 中处理。

这样虽然每个 Provider 都需要维护自己的请求逻辑，但整体结构更加清晰。

---

## 如何减少重复代码

随着支持模型数量增加，我开始思考哪些代码是真正可以复用的。

最终，我将：

- 模型管理；
- Session 管理；
- 数据管理；

都进行了独立封装，而每个 Provider 只负责自己独有的实现。

这样既保留了不同模型之间的差异，也减少了重复代码。

---

## 如何让业务层更加简单

我希望 ChatServer 的代码尽可能简单。

因此，大部分业务逻辑都放在 ChatSDK 内部。

ChatServer 更多扮演"调用者"的角色：

接收请求 → 调用 SDK → 返回结果。

这种分层方式也让我体会到 SDK 的意义不仅是代码复用，更重要的是隐藏实现细节。

---

# 项目收获

完成整个项目后，我最大的收获主要来自三个方面。

## 工程组织能力

相比以前完成的小型练习，这个项目涉及多个模块之间的协作。

在开发过程中，我开始尝试按照职责划分模块，而不是把所有代码集中在一个类中。

虽然还有很多可以改进的地方，但也让我逐渐建立了工程化开发的意识。

---

## 阅读和理解第三方库

项目中使用了多个第三方库，例如：

- cpp-httplib
- JsonCpp
- SQLite3
- spdlog

在学习它们使用方式的过程中，我也逐渐习惯了阅读官方文档和示例代码，而不是只依赖网络教程。

这对后续学习新的 C++ 库也很有帮助。

---

## 从"写代码"到"设计代码"

以前更多关注代码是否能够运行。

而在完成这个项目之后，我开始尝试思考：

- 为什么这样划分模块？
- 为什么接口要这样设计？
- 为什么需要隐藏实现细节？

虽然目前理解还比较浅，但我认为这种思考方式比单纯实现功能更加重要。

---

这些也是我后续继续学习 C++ 工程开发时，希望不断改进的方向。
